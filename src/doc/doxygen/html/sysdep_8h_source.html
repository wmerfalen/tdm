428 /* Function prototypes ************************************************/
429 
430 /*
431  * For reasons that perplex me, the header files of many OS's do not contain
432  * function prototypes for the standard C library functions. This produces
433  * annoying warning messages (sometimes, a huge number of them) on such OS's
434  * when compiling with gcc's -Wall.
435  *
436  * Some versions of CircleMUD prior to 3.0 patchlevel 9 attempted to
437  * include prototypes taken from OS man pages for a large number of
438  * OS's in the header files. I now think such an approach is a bad
439  * idea: maintaining that list is very difficult and time-consuming,
440  * and when new revisions of OS's are released with new header files,
441  * Circle can break if the prototypes contained in Circle's .h files
442  * differs from the new OS header files; for example, Circle 3.0
443  * patchlevel 8 failed with compiler errors under Solaris 2.5 and
444  * Linux 1.3.xx whereas under previous revisions of those OS's it had
445  * been fine.
446  *
447  * Thus, to silence the compiler warnings but still maintain some level of
448  * portability (albiet at the expense of worse error checking in the code),
449  * my solution is to define a "typeless" function prototype for all problem
450  * functions that have not already been prototyped by the OS. --JE
451  *
452  * 20 Mar 96: My quest is not yet over. These definitions still cause
453  * clashes with some compilers. Therefore, we only use these prototypes
454  * if we're using gcc (which makes sense, since they're only here for gcc's
455  * -Wall option in the first place), and configure tells gcc to use
456  * -fno-strict-prototypes, so that these definitions don't clash with
457  * previous prototypes.
458  *
459  * 4 June 96: The quest continues. OSF/1 still doesn't like these
460  * prototypes, even with gcc and -fno-strict-prototypes. I've created
461  * the constant NO_LIBRARY_PROTOTYPES to allow people to turn off the
462  * prototyping.
463  *
464  * 27 Oct 97: This is driving me crazy but I think I've finally come
465  * up with the solution that will work. I've changed the configure
466  * script to detect which prototypes exist already; this header file
467  * only prototypes functions that aren't already prototyped by the
468  * system headers. A clash should be impossible. This should give us
469  * our strong type-checking back. This should be the last word on
470  * this issue!
471  */
